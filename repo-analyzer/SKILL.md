# Repo Analyzer

Deep-scan a codebase and produce the foundational learning artifacts. This is the
most token-expensive skill in the suite — it reads most of the codebase before
generating anything.

## Analysis Flow

Follow the global→local cognitive flow identified in codebase comprehension research
(Gao et al., 2025 — CodeMap):

### Level 1: Global (Project Overview)
- Read README, docs/, CONTRIBUTING, build config
- Identify: project purpose, target users, key dependencies, ecosystem position
- Output: `learn/overview.md`

### Level 2: Structural (Architecture)
- Map module/package structure via directory tree + imports
- Identify: core abstractions, data flow, entry points, extension points
- Read tests directory for coverage map
- Output: architecture section of curriculum

### Level 3: Local (Functions and Variables)
- Deep-read public API, key internal modules, examples
- Identify: function signatures, patterns, gotchas, underdocumented behavior
- Output: detailed curriculum sections, cheatsheet, quiz bank

## Output Artifacts

### curriculum.md
Phased learning path:
- Phase 1 (Usage): how to use the library
- Phase 2 (Internals): how it works under the hood
- Phase 3 (Contribution): how to extend or modify it

**Every section MUST have an explicit HTML anchor** above its heading for stable
cross-linking from README.md and path.html:

```markdown
<a id="s1.1"></a>
### Section 1.1: [Title]
**Goal:** [one sentence]
**Read these files:** [specific paths with line ranges and focus notes]
**Key concepts:** [2-4 bullets]
**Watch out for:** [gotchas, bad naming, missing docs]
**Self-check:** [2-3 questions answerable from the reading]
**Depends on:** [previous section IDs]
```

Do NOT rely on GitHub's auto-generated anchors — they break across viewers.

### learn/README.md (THE entry point)
Phase-grouped table with markdown checkboxes. One row per curriculum section:

```markdown
# Learning Path: [Project Name]

> Generated by repo-learner. Open this alongside your editor and `/learn` chat.

## Phase 1: Usage

| | Section | Read | Do | Test |
|---|---|---|---|---|
| [ ] | 1.1 [Title] | [curriculum](curriculum.md#s1.1) | [exercise](notebooks/exercise-01-...) | `/learn quiz 1.1` |
| [ ] | 1.2 [Title] | [curriculum](curriculum.md#s1.2) | — | `/learn quiz 1.2` |
```

Sections without exercises get "—" in the Do column. The orchestrator parses these
checkboxes for progress state.

### learn/path.html
Single self-contained file. Interactive checkboxes persisting to localStorage. "Export
progress" button copies updated markdown to clipboard for round-trip paste into README.md.
No server, no build step. Generate per-curriculum during the pipeline.

Use `scripts/scaffold_path_html.py`. Construct a `PathSpec` (project name, intro, phase
names, list of `PathSection` rows mirroring the curriculum sections) and call
`generate_path_html(spec, "learn/path.html")`. Or write a JSON spec and invoke as a CLI:

```
python scripts/scaffold_path_html.py --spec path_spec.json --output learn/path.html
```

Section IDs (e.g. `1.1`) map directly to the explicit `<a id="s1.1">` anchors in
`curriculum.md` — the script defaults the read URL to `curriculum.md#s{id}`. Notebooks
are passed as `(label, href)` pairs; empty list means no exercise for that section.

### overview.md
1-2 page big-picture document: what, who, why, how, prerequisites, architecture sketch.

### cheatsheet.md
Under 200 lines. Import patterns, common usage snippets, key API reference, gotchas.

### learn/internals/quiz-bank.md
3-5 questions per curriculum section, tagged by difficulty (warm-up / conceptual / hard).
Every question includes a code snippet for context. Answers behind `<details>` tags.
Lives in internals/ — this is a working document that the quiz skill can read AND edit.

## Quality Standards
- File paths in curriculum must be verified to exist
- Self-check questions must be answerable from the cited reading
- Cheatsheet snippets must be syntactically correct
- Be honest about code quality — flag confusing, underdocumented, or poorly designed areas
